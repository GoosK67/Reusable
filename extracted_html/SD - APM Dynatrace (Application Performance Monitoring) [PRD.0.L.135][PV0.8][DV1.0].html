Service Model

Technical Application Management (TAM)

This service is dependant of other layers (IaaS and/or PaaS) which are indicated in the ‘Obligatory’ chapter of this document and which have their own Service Description. The combination of the layers assures the customer of the ‘infrastructure up to the technical application’ management. These obligatority services are called the ‘Platform’.

Service summary

Technical Application Management

Application Performance Management (Observability) is a Service which foresees in the monitoring of the performance of specific business (critical) applications and its end2end flows. This service is based on the Dynatrace Software Intelligence Platform.

The customer can choose out of a range of modular components within Dynatrace which can build up to a full scope APM service.

The Service provides the tasks and activities needed to manage an Application Performance Management environment using Dynatrace (The Application) from a technical perspective. Together with the underlaying obligatory platform services, Cegeka unburdens the customer with all technical aspects from infrastructure up to the technical Application.

The Standard Service features are focussed to keep the Service up-and-running, to keep it secure and to optimize its technical parameters.

The Application

The Dynatrace solution consists out of the following components:

  • Dynatrace Server
    • The Dynatrace Server or Dynatrace cluster node is the heart of this monitoring environment. It provides the user with an UI and all necessary components to evaluate, store and process monitoring data. This server can be set up as a cluster in order to maintain continuous monitoring even during downtime of one of the cluster nodes. The number of nodes is defined based on several parameters as well as on Dynatrace recommendations/best practices.
  • ActiveGate’s
    • An ActiveGate works as a proxy between the monitoring agents, external services and the Dynatrace Server. Dynatrace Managed deployments typically require one or more Environment- and/or Cluster ActiveGate’s. The number of required ActiveGate’s will be determined based on the services used by the customer as well as the availability requirements.
  • OneAgent
    • The Dynatrace OneAgent monitors the performance of the hosts in your application-deployment environment. They way the monitoring is performed is dependent on the underlaying mode.

The Dynatrace solution is hosted within the Cegeka datacenters in Hasselt & Geleen and has a permanent uplink with Dynatrace Mission Control in Ireland. Dynatrace Mission Control provides fully automated management capabilities in a proactive way that keep Dynatrace server secure, reliable, and up-to-date—all while saving the user from the hassles of administrative tasks like upgrades and troubleshooting. Dynatrace product experts can remotely assist with upgrades and troubleshooting when you run into problems.

Service features

Standard Service features

Application knowledge

Cegeka ensures skilled engineers with experience on the Dynatrace technology in order to perform the Service.

Technical setup

Cegeka advises in the setup of the underlaying platform in order to set it up according to the technology requirements of the Application. This may be different for each Customer environment (DTAP - Development, Test, Acceptance, Production and it’s variants).

High availability setup: this setup advise includes also the setup of HA (High Available) environments where needed. Cegeka will advise a cost effective solution within the context of the available platform(s) of the Customer and the best practices of the Application:

  • Dynatrace Managed Cluster
    • HA is foreseen using a multi-node cluster consisting of at least 3 nodes for high availability and data redundancy. Additional nodes may be deployed for horizontal scaling, up to the allowed maximum of nodes.
  • ActiveGate
    • HA is available for both DMZ and LAN and is provided by multiple ActiveGate’s located in both datacenters. ActiveGates in Azure are the responsibility of the Customer. However Cegeka will inform the Customer when irregularities with them are detected

Cegeka performs and maintains the technical installation of the Application on the agreed platform ensuring its correct working.

Technical setup is documented on operational level and is available to the Customer.

Disaster Recovery

Disaster Recovery is available for the Dynatrace Managed platform and is based on two mechanisms:

  • The Dynatrace Managed Cluster, which is used for both production and nonproduction, resides in the foundation environment and is hosted on TIER 1 virtual machines (includes DR to the secondary datacenter)
  • PRD & non-PRD ActiveGate’s are spread over both datacenters and are operating in an active-active setup

Technical management

  • Cegeka is responsible for the technical operation of the Application and to assure it’s correct running during the agreed availability windows.Customer / Cegeka meetings

User group meeting (operational meeting)

Customer foresees a recurrent user group meeting where Cegeka amongst others will:

  • Report on new platform features
  • Give general recommendations
  • Elaborate on use cases
  • Discuss known issues
  • Discuss open Changes
  • Interaction with owners of monitored applications (functional) regarding:
  • Added or changed monitoring (as result of requests)
  • Measures were taken to tune out false alerts
  • The noisiest application and alerts
  • Participants of this meeting are: Cegeka APM engineers and Customer APM engineers and (optional) owners of monitored applications
  • The meeting schedule is 1x/month

Dynatrace Customer Success meetings (operational meeting)

Cegeka will join the recurrent Argenta & Dynatrace Customer success meetings.

  • Report issues for which we need Dynatrace engineers
  • Escalate problems
  • Participants of this meeting are: Dynatrace customer success manager (organiser), Cegeka APM engineers, Customer APM engineers
  • The meeting schedule is 1x/month

Housekeeping

Recurrent maintenance activities (housekeeping) are needed to ensure the correct working of the Application. Automation is applied where possible.

Proactive management

Housekeeping includes following regular checks:

  • The setup and the running of the Application on its platform
  • System trends (trends of CPU/memory/disk usage) on deviations in evolution
  • Perform and check log rotation (cleanup of older logs)
  • Assure application settings are part of the backup of the underlaying platform
  • Resolution actions are triggered when needed and followed to assure the SLA.

Patching

See Life Cycle Management. Dynatrace is continuously updated.

Patching of the underlying platform is part of the Service of that platform.

Dynatrace follows an agile development process with sprints every two weeks where every 2nd sprint is publicly released. These releases are then supported for:

  • Dynatrace server – 3 months
  • ActiveGate – 9 months
  • OneAgent – 9 months

This frequent release process demands continuous lifecycle management where updates will be deployed at the following frequency:

  • Dynatrace server – monthly
  • ActiveGate – monthly
  • OneAgent – separate planning based upon agreed maintenance windows as this requires restarts of the business applications.

Cegeka is fully responsible for the patch process and all patching activities will follow the change management procedures in place.

Life Cycle Management

Cegeka monitors the Technology version of the service and notifies the Customer in accordance with the agreed LCM policy (see cross-functional Services) when used versions are subject to end-of-support or end-of-life. Appropriate actions (updates, replacement or other actions) are planned in accordance with agreed maintenance windows and will be executed on T&M basis in case of major updates. Patches and minor updates are included in the service cost as part of ‘Patching’.

Monitoring

Cegeka monitors the correct working of the Application on all Customer environments.

  • This is done by means of scripts (with API calls) to check the correct technical application functionality.
  • The checks are executed every 3 minutes.

Monitoring improvements will be done during the lifecycle of the Service and are based on evaluation of incidents, problem management or other suggestions.

The underlaying platform is also monitored according to the related Service Description of the platform.

Monitoring events are evaluated during the Service Window in accordance to the Service Level of the individual environment. Notable events are treated as Incident tickets and are handled according to regular incident management.

Also, a specific Dynatrace monitoring profile to monitor the availability of the platform itself, this profile includes:

  • Process checks
  • Port checks
  • Database replication status
  • Verification of cluster and ActiveGate health using REST API’s

Security and Compliance

Management of technical users

The management of technical users (for engineers) is done by Cegeka in agreement with the Customer and in accordance to the Customer IAM policies.

Security Management

The security management is performed following the Cegeka internal security policies and the contractual Customer security policies.

Security incidents are created in case a security breach is detected and is immediately communicated to the Customer and Cegeka stakeholders.

SIEM

Additional monitoring can be done based on System and/or Application logs, which can be forwarded to a Customer SIEM solution (Security reason) or other solution (compliance reason, e.g. central collection of application logs to oversee the application stack) on request.

Underlaying Platform hardening

The underlaying platform and OS will be hardened as part of and according to the Service Description of the related Services.

Backup and Restore

The operational Backup Plan contains the information on frequency and retention of the backups in accordance with the Customer backup policy.

Restores can be done both on file and on full system level. These can be requested via the regular RFC process.

Duration of a restore is dependant of the request (file or full; size of the item(s) to be restored).

Application settings are subject to the backup of the underlaying platform.

All the important Dynatrace Managed configurations and monitoring data will be backupped.

For the avoidance of doubt, the following items are excluded from backup:

  • Users sessions (due to GDPR compliancy)
  • Raw monitoring data / transaction storage
  • Timeseries data: 1 minute and 5-minute resolution data
  • Backup logs gathered by Log Management (logs from applications are backupped on the source systems)

Incident and problem management

Besides monitoring incidents (see ‘Monitoring’) also other incidents can be reported by both the Customer and Cegeka. These will be subject to the regular incident management process (See Cross-functionals).

Related to incident management, Cegeka performs also problem management (See Cross-functionals).

If an incident cannot be resolved by Cegeka’s engineering support, due to for example a bug in the software, Cegeka will make use of the Argenta support contract and log a case with Dynatrace.

Resolution times of incidents where support is needed from Dynatrace are subject to the selected support contract.

Customers responsibility

The customer is responsible for defining the monitoring needs and specifying how the application should be monitored.

  • For Dynatrace, this includes determining the specific functional application monitoring needs and Cegeka will translate them in configurations within the platform

Requests

Both the Customer and Cegeka can issue a Request for Change (RFC) or a Request for Information (RFI).

There are multiple types of requests:

  • Service Request
    • This is a pre-defined request
    • Some can be requested via Horizon: these are pre-approved and pre-defined requests
    • Service request that is not pre-approved will be executed after approval by the CAB
    • Service Requests are executed as a change ticket and will be delivered within the indicated lead time.
    • The pre-defined execution is optimal automated.
    • This is used for requests that are often needed (multiple times per month).
    • These have a fixed price.
  • Normal Changes:
    • These are other non-pre-defined requests
    • They need an evaluation and analysis where Cegeka needs to deliver a proposal and impact analysis to address the change request (with a T&M estimation).
    • A successive review, approval and planning (maintenance window) by a CAB of the proposal is needed prior to execution.
    • These are billed on T&M basis.

The handling of Requests is subject to the regular change management process (see Cross-functionals).

RACI role definition:

  • R-Responsible: The party who is responsible for the execution of the activity.
  • A- Accountable: The party ultimately accountable for the correct and thorough completion of the task and/or activity as performed by the Responsible. For each task and/or activity exactly one Accountable must be specified. The Accountable has the mandate to accept/dismiss on work provided by the Responsible. The Accountable is by default reactive.
  • C-Consulted: The party who must be consulted and must approve or provide input prior to execution. This party (partly) gives direction to the result. This is about two-way communication.
  • I-Informed: That is the person/party who is informed about the decisions, progress and results achieved, so that the next step can be taken. This is about one-way communication.
