Service Description

Service summary

With Automation as a Service we focus on automating repetitive tasks or services to improve efficiency and reduce errors. From a technology point-of-view we use the Cegeka Hybrid Cloud standard software tools to streamline processes and enhance service delivery across various sectors.

Key aspects of Service Automation

  • Purpose: Service Automation aims to increase efficiency, improve accuracy, enhance customer experience, reduce costs, and provide scalability.
  • Components:
    • Process Definition: Documenting steps involved in a workflow
    • Workflow Management: Monitoring and controlling task flow
    • Task Automation: Automating individual tasks within the workflow
    • Integration: Connecting with other systems and data sources
    • Reporting and Analytics: Tracking and analyzing performance metrics
  • Applications:
    • ICT Automation: Includes configuration management, self-service, workflow automation, monitoring and maintenance and syncing & provisioning.
  • Benefits:
    • Reduced errors and improved accuracy
    • Time and cost savings
    • Enhanced resource allocation
    • Improved scalability
    • Increased customer satisfaction

Services

Standard Services

Provide automation support for the HC and DWP competences from ideation over analysis & design to implementation phase

Automation support has to be considered on different levels and aspects:

  • Implementing service requests and link with Horizon
  • Providing shared libraries: global (supported within automation) vs team (supported within the CC’s)
  • Assisting in and/or creating of Jenkins pipelines: integration to Cegeka tooling (Horizon/USD/etc), while the CC can focus on their specific technical pipeline
  • Oros reporting & dashboarding
  • Scheduling support with supported schedulers (at this moment: IWS) (enterprise grade + low-level tool)
  • Framework & guidelines usage of Jenkins
  • etc

Automation requests are collected & consolidated based on the input of the other competences (technical CC’s within delivery, change management & architecture)

  • Improving existing automations
  • Designing and implementing new automations in close cooperation with the other competences for technology-specific coding
  • Functional analysis will be foreseen from within the automation team

RACI Role definition

  • Responsible: The party who is responsible for the execution of the activity.
  • Accountable: The party ultimately accountable for the correct and thorough completion of the task and/or activity as performed by the Responsible. For each task and/or activity exactly one Accountable must be specified. The Accountable has the mandate to accept/dismiss on work provided by the Responsible. The Accountable is by default reactive.
  • Consulted: The party who must be consulted and must approve or provide input prior to execution. This party (partly) gives direction to the result. This is about two-way communication.
  • Informed: That is the person/party who is informed about the decisions, progress and results achieved, so that the next step can be taken. This is about one-way communication.
