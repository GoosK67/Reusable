Service description

Service summary

The CPC Power AIX PaaS service is the platform-based solution that virtualizes and brings the customer’s Power AIX environment to the cloud using a scalable model based on IBM power systems.

Service description: features and functionalities

This service provides Power AIX based Lpars for which initial setup and recurring services are provided. Supported operating system is 7.1 or higher (*).

It gives your business peace of mind with disaster recovery and high availability solution (HA/DR). This includes cold and warm off-site recovery depending on the scenario adopted.

Two types of deployments are provided:

  • Single Lpar is offered as a non-redundant platform service were customer data and system configurations are protected by local backup procedures
  • Dual Lpar service is a dual architecture that provides redundancy across location. On top of this, local backup procedures provide an additional protection of the customer’s environment.

(*) Although AIX 6.1 is out of supported by IBM since May 2017, we can provide support (Base Level 6100-09-03) on best effort for legacy AIX environments.
AIX 7.1 is supported on IBM Power 9 systems or older.

CPC Power AIX PaaS service description

Single Lpar build

The setup of a single Lpar on a power system including following building stones (in this scenario there is no high availability Lpar provided to ensure redundancy)

  • Single Prod Compute (CPU)
  • Single Memory (RAM)
  • Single external storage: production (Tier 0 or Tier 1)
  • DC replicated backups (local backup volumes with automatic replication to a remote Cegeka data center)
  • Platform management & monitoring services
  • Monitoring of major OS parameters (DASD, CPU, memory, …)
  • Monitoring of the processes and services required by the customers (might incur additional charges)
  • Customer notification (phone call/mail/SMS) in case of priority 1 alerts

Dual Lpar build:

Setup up of a primary Lpar along with a 2nd Lpar to ensure high availability.

  • Dual compute (CPU)
  • Dual memory (RAM)
  • Dual storage (Tier 0 or Tier 1)
  • DC replicated storage (local storage volumes with automatic replication to a remote Cegeka data center)
  • DC replicated backups (local backup volumes with automatic replication to a remote Cegeka data center)
  • Platform management & monitoring services
  • Monitoring of major OS parameters (DASD, CPU, memory, …)
  • Monitoring of the processes and services required by the customers (might incur additional charges)
  • Customer notification (phone call/mail/SMS) in case of priority 1 alerts
  • Connectivity between the 2 locations required for storage replication
  • Replication license required for replication between both Lpars

Set up activities

  • Setup backup: installation and configuration of backup application including documentation
  • Setup monitoring of major OS parameters (CPU, memory, filesystems …)
  • Setup (customer) procedures that need to be executed when a monitoring alert is generated
  • Update CMDB: Adding the newly created Lpars to the company management database and keeping it up to date

Operational services

Incident & Problem Management

  • Reactive support

    • Receive, track and resolve incident tickets.
    • Identify the cause of an incident.

Availability Management

This service is based on the Cegeka Cloud platform which has the following availability targets:

  • 99,95% for a single instance infrastructure component deployed in 1 Cegeka datacenter
  • 99,99% for dual instance infrastructure components, deployed across 2 Cegeka datacenters

Above targets apply as SLA for the operating services running on the Cegeka power cloud platform.

For services running on customer premise or in the public cloud these targets apply as SLO (Service Level Objectives) as Cegeka is not responsible for the underlaying infrastructure.

Backup, Restore, Disaster Recovery

Following features are included in this service

  • Switch test (only for dual Lpar): failover test from primary to secondary Lpar
  • BCRS test (restore test from scratch on different Lpar)

User & Access Management

The Cegeka CPC team members need to have credentials with (adopted) full rights, in order to allow to setup, configure, and administer the Lpar. As part of standard changes, users and accesses can be created, modified or removed by Cegeka. Administrator account will always be managed by Cegeka

Patching

Patching of the Power AIX Lpars is defined at the start of the contract with a mandatory minimum of once per year for compliance reasons.

Lifecycle Management (LCM)

CPC lifecycle management will follow the IBM release life cycle for the OS installed on the Lpar. IBM will announce end of program support date with at least 12 months’ notice prior to the effective termination date. Customers with Lpars that are in the last 12 months of the support cycle will be informed and encouraged to upgrade to more recent OS version. Customer not in a position or willing to upgrade can make use of the IBM service extension offering. On expiration of this extended offering, customers will still have to upgrade or sign a risk waiver. For the hardware lifecycle management, we refer to the underlaying IaaS service description.

Security

Cegeka has defined a security baseline. It describes the Cegeka recommended security rules to achieve an appropriate level of security on Power systems and the CPC environment. The result is the balance between the vendor and Cegeka recommendations and what is practically feasible.

Customers can, in consultation with Cegeka, have their additional security baselines implemented on request and at additional cost. However, these must not prevent Cegeka from properly managing the systems or from fulfilling its service responsibility on the Lpar.

Capacity Management

Capacity management of the CPC environment is included in this service in order to guarantee necessary resources for current and future business.

In this regard, Cegeka is entitled to switch LPARS between Power servers and customer needs to apply for new license keys if necessary.

Customers are informed about capacity related activities only if customer impact is expected.

Disaster recovery

RPO

  • Single setup systems have an RPO of max 24 hours, based on the daily backup time schedule and the fatal disruption moment.
  • Dual setup systems have an RPO of last transaction in case of a consistence database on the production environment. If, however, the data is inconsistent at the moment of disruption, we need to rely on the daily backup schedule that could result in a RPO of 24 hours.

RTO

For service levels that include dual setup systems, the target system will be available within the timeframe of the previous switch test time plus 50%.

Service levels that include single setup systems have an RTO that is equal to the time of the yearly restore test plus 50%.
