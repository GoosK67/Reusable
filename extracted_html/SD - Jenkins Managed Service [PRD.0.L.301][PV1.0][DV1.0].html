Definition

Jenkins is an open-source automation server used for continuous integration (CI) and continuous delivery (CD). It helps automate various stages of the software development lifecycle, including building, testing, and deploying applications.

Service Description

The Jenkins Managed Service, can be quoted or ordered using the following product SKU's :

  • SKU code : 0.L.301.001 Jenkins - Namespace Deployment (one-off)
  • SKU code : 0.L.301.002 Jenkins - Extra Customer / Agent Configuration (one-off)
  • SKU code : 0.L.301.003 Jenkins - Namespace Management (recurring)

Service summary

This document describes the services the Open Systems team provides for the Jenkins Managed Service offering.

Services

Standard Services

Generic description

The Open Systems team is committed to ensuring the availability of Jenkins services within the shortest possible timeframe, adhering to SLA, RTO, and RPO targets, while considering the inherent limitations of the team. All activities described below are part of the management of the service and directly linked to the Jenkins - Namespace Management (recurring) - SKU, which is billed on a recurring monthly basis.

We will install and configure the Jenkins infrastructure to ensure it is ready for end-user use. However, any configuration within the application itself remains the responsibility of the end user. Our primary objective is to provide technical application support, which includes regular updates and patch management. Major version upgrades, however, are excluded from the scope of standard support and can be carried out under a separate, billable Request for Change (RFC).

For scheduled activities, predefined service windows will be utilized. During these periods, Jenkins may experience temporary unavailability. All planned interruptions will be communicated in advance through the built-in banner feature within Jenkins. Note: theese SLA's are not applicable for bugs in community supported tools (Jenkins).

Backup and recovery

Each Jenkins instance is backed up using an Image-Level Backup solution in accordance with the “Cegeka - O.Open Systems - Preferred 1 month replicated” backup policy.

Monitoring and event resolution

Each Jenkins instance is monitored using standard Cegeka monitoring tools, which include:

  • the default Linux monitoring template
  • port monitoring
  • login page health check

Instances covered under this service are subject to both reactive and proactive monitoring. This is based on various criteria and triggering events, which may result in incidents being assigned to the Open Systems team. Monitoring-related incidents are managed according to their priority and within the designated service windows.

Security baseline compliancy

As with any server provided by our team, each Jenkins instance will have the latest approved Cegeka security baseline applied during the initial setup. Updates to the security baseline will be implemented via a Request for Change (RFC) when necessary.

Capacity Management

Cegeka ensures operational capacity monitoring for storage and compute resources through monitoring events. When predefined standard thresholds are exceeded, monitoring incidents are triggered with the appropriate priority for further action.

Lifecycle Management

Both the operating system and the Jenkins service will be patched regularly in alignment with Open Systems standards. Patching will follow the established cycles used for all managed servers and adhere to the DTAP (Development, Testing, Acceptance, Production) process.

For Jenkins, the latest stable version will always be maintained. Minor and major updates will be applied systematically, ensuring that outdated versions of Jenkins are never in use.

User Management

User management is conducted in compliance with the prevailing security baselines, which include the following measures:

  • access is only allowed with a Cegeka local admin account
  • MFA is enforced
  • only Cegeka Hybrid Cloud teams can get access by default
  • other Cegeka teams can be allowed access when purchasing the service

Every new users access needs to be approved by an administrator. Access and permissions will be granted based on the engineers resolver group.

Jenkins environment

Within the Jenkins instance, each Competence Center (or resolver group, if no Competence Center is defined) is provided with an isolated environment. A recommended setup is outlined in our Best Practice - Jenkins guidelines1. While adherence to these recommendations is not mandatory, deviations may limit the availability of optional services.

Outside the designated Competence Center group, creating additional groups is not permitted, ensuring enforcement of the predefined setup.

1 the corresponding document is stored in the Confluence space of the Open Systems team and if needed can be requested through the DA.

Liability

As Jenkins is an open-source tool, we cannot guarantee any SLA, RTO, or RPO commitments if the root cause of faulty behavior is attributable to a bug within the Jenkins software.

Although this type of request is rather exceptional are rarely applies, the creation of additional service accounts can/should be requested/raised by creating a support ticket, as there is no automation supporting it.

RACI Role definition

  • Responsible: The party who is responsible for the execution of the activity.
  • Accountable: The party ultimately accountable for the correct and thorough completion of the task and/or activity as performed by the Responsible. For each task and/or activity exactly one Accountable must be specified. The Accountable has the mandate to accept/dismiss on work provided by the Responsible. The Accountable is by default reactive.
  • Consulted: The party who must be consulted and must approve or provide input prior to execution. This party (partly) gives direction to the result. This is about two-way communication.
  • Informed: That is the person/party who is informed about the decisions, progress and results achieved, so that the next step can be taken. This is about one-way communication.
